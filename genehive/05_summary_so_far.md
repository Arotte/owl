I've read through all four files (the summary, the budget-control research, the MWE design conversation, and enough context from the HTML system doc). Here's my understanding of Genehive:

**The core idea.** Genehive is a general-purpose framework for autonomous codebase evolution. You point it at a git repo, give it a goal and one or more eval commands, and it runs a closed loop overnight: mutate the code, run the evals, judge whether the mutation is better, keep or revert, repeat. It generalizes Karpathy's `autoresearch` from a one-off experiment into a tool any developer can use on any codebase — the genome is your code, the phenotype is what it does when run, fitness is whatever your eval commands measure.

**The conceptual frame.** The design is built on a biology metaphor that isn't just decoration — it shapes the architecture. `autoresearch` is recognized as a hill climber with an LLM as the mutation operator, missing crossover. The full vision adds a cast of named agents (Wallace the orchestrator, Mendel the mutator, Cupid for crossover, Reaper the judge, Fossil the historian, Hermes for cross-island migration) and an island-model population structure, which evolutionary graph theory says is optimal for rugged fitness landscapes. There's also a layered eval curriculum (smoke test → proxy sprint → capability unlocks → blended pressure → integration oracle) inspired by curriculum learning, with stage transitions triggered by convergence rather than wall time.

**The MWE that's actually being built.** Everything above is the long-term shape; the starting point is dramatically smaller. One island, one worktree, one branch — a hill climber, not a population. No SQLite and no Fossil: git *is* the database, with mutation reasoning, eval output, and judge verdicts stored in commit messages, and `git revert` as the cull mechanism. Three cleanly separated stages per iteration: Mutate (a full Cursor CLI agent session), Evaluate (plain shell subprocesses, no LLM), Judge (a single cheap structured LLM call that sees the new eval output vs. the previous best and returns KEEP or CULL). The mutator never judges its own work; the judge never sees the code. Config lives in a `.genehive/` directory in the target repo.

**The UX.** A small `gene` CLI — `init`, `check`, `start`, `watch`, `stop`, `status`, `report` — with `gene watch` as a polished TUI dashboard over the git log. The guiding principle is zero friction: `pip install`, `gene init`, `gene start`, walk away.

**One unresolved tension I noticed.** File 02 argues convincingly that Claude Code is strictly better than Cursor CLI for this use case because it exposes `total_cost_usd` per call, giving exact budget control instead of `tiktoken` lower-bound estimates. File 04, written later, explicitly defers that migration and sticks with Cursor CLI + `tiktoken` for v0. That's a reasonable call for getting something working, but it does mean the MWE ships with the weaker budget story that file 02 was written specifically to fix — worth flagging as the first thing to revisit once the loop is green.

Overall: the idea is clean, the biology metaphor genuinely earns its keep by suggesting concrete architectural moves (crossover, islands, curriculum), and the MWE scoping is disciplined — git-as-database and the mutator/evaluator/judge split are particularly elegant simplifications.
